# Mail platform (team-managed email and transactional delivery)

## Purpose and positioning

DeployWerk already owns **domains** (with TLS management as specified elsewhere), **servers** (SSH-accessible hosts with egress), and **secrets** (encrypted credentials). A team that deploys applications on its own infrastructure should be able to run **email for its own domains** and **transactional mail for its applications** without mandatory reliance on external SaaS providers (SendGrid, Mailgun, SES, etc.).

This specification describes a **first-class mail platform** integrated with the DeployWerk control plane. Implementation may be phased; release criteria are in [Phased delivery](#phased-delivery-release-criteria).

**Related:** [01-actors-identity-orgs.md](01-actors-identity-orgs.md) (identity), [02-projects-through-applications.md](02-projects-through-applications.md) (applications, destinations), [04-security-rbac-secrets-compliance.md](04-security-rbac-secrets-compliance.md) (secrets, audit), [07-operations-enterprise.md](07-operations-enterprise.md) (notifications, health checks). Implementation status and gaps: [../STATUS.md](../STATUS.md), [../ENTERPRISE_GAPS.md](../ENTERPRISE_GAPS.md).

## Architecture overview

Within a team workspace:

- **Mail domains** — MX, SPF, DKIM, DMARC (and optional BIMI); distinct from HTTP routing domains though a hostname may serve both.
- **Mailboxes, aliases, rules, mailing lists** — addressing and routing policy.
- **Managed mail server** — [Stalwart Mail Server](https://stalw.art/) (SMTP, IMAP, JMAP) as a **`mail_server` managed application type**: configuration generated by DeployWerk, upgrades via the normal deploy pipeline, admin API reachable only through the control plane (e.g. SSH tunnel), not the public internet.
- **Webmail** — Built into the DeployWerk web application; primary protocol **JMAP** (RFC 8620/8621) for efficiency; IMAP optional for external clients.
- **Transactional mail API** — REST API for applications to send mail (templates, suppressions, delivery status).

Integrations: team **secrets** (DKIM private keys, relay passwords, TLS material, app passwords), **deploy pipeline** (mail server version, config pushes, DKIM rotation), **notifications** (instance email channel may route through this platform when enabled), **RBAC** and **audit** (see [Security and governance](#security-and-governance)).

**Implementation note:** TLS for mail may follow the same ACME/Let's Encrypt trajectory as application domains in [04](04-security-rbac-secrets-compliance.md). Until ACME is fully productized for all surfaces, operators may install certificates via team secrets and documented procedures; the spec does not require a particular rollout order.

## Mail domains

### Functionality

- Register one or more **mail domains** per team.
- **DNS guidance wizard:** after a domain is added, the UI shows required DNS records and validates them with live lookups before activation.
  - **MX** — points to the team’s mail destination (inbound).
  - **SPF** — e.g. `v=spf1 include:<instance-domain> ~all` or team-defined policy.
  - **DKIM** — DeployWerk generates key material (e.g. RSA-2048 or Ed25519); private key in team secret store; UI shows **TXT** for the public key. Multiple **selectors** for rotation.
  - **DMARC** — wizard guides policy progression (`none` → `quarantine` → `reject`); optional aggregate report receiver (including DeployWerk-internal parsing in later phases).
  - **BIMI** — optional brand indicator (VMC, SVG logo) in advanced phases.
- **DKIM rotation** — new selector, overlap period (default e.g. 7 days), retire old selector.
- **Domain verification** — dashboard shows per-record pass/fail/propagating.
- **Catch-all** — configurable target or discard for unmatched local parts.
- **Multi-domain** — one Stalwart instance may serve multiple domains; per-domain DKIM selectors and DMARC policy.
- **Send-only domains** — outbound transactional use may omit MX; SPF + DKIM suffice.

## Mail server provisioning (`mail_server`)

### Behaviour

- When a team enables mail, DeployWerk provisions Stalwart on a **team-chosen destination** (same deployment model as other applications).
- **Configuration is generated** from mail domains, mailboxes, and routing records; operators do not hand-edit Stalwart config files in the common case.
- **DKIM** private keys are injected at runtime via `dw_secret:NAME` (or equivalent secret references).
- **Stalwart admin API** — bound to localhost on the host; DeployWerk uses **SSH tunnel** (or equivalent secure channel) from the control plane; not exposed on the public network.
- **Updates** — new Stalwart versions ship through standard deploy jobs; teams opt into upgrades in the UI.

### Provisioning flow (normative intent)

1. Team admin enables mail, selects destination.
2. **Preflight:** port 25 reachability (e.g. probe to a well-known MX), PTR/DNSBL checks as specified in [Spam, reputation, and abuse](#spam-reputation-and-abuse). If port 25 is blocked, surface provider-specific guidance and **relay mode** (outbound via upstream SMTP).
3. DeployWerk generates encrypted application config; injects secret references.
4. Deploy job starts the mail container; health verified (e.g. Stalwart admin health).
5. DNS wizard; when MX (and required records) validate, domain status becomes **active**.

### Relay mode

Many hosts block **outbound port 25**. The platform MUST support routing outbound mail through a configurable **SMTP relay** (credentials in team secrets) while keeping inbound and webmail self-hosted where the architecture allows.

## Mailboxes, aliases, and lists

### Mailbox types

- **Personal** — linked to a DeployWerk user; webmail uses DeployWerk session.
- **Shared** — multiple members; shared visibility policy.
- **Service** — automation/application; credentials in secret store.
- **Catch-all** — domain-level unmatched local parts.

### Other objects

- **Aliases** — local-part maps to one or more mailboxes or external addresses.
- **Mailing lists** — moderation, roster, `List-Unsubscribe` (later phases).
- **Forwarding rules** — per-mailbox forward to address, mailbox, or webhook.
- **Auto-reply** — time-bounded OOO or acknowledgement.
- **Quotas** — per-mailbox and team aggregate; soft/hard limits and alerts.
- **Retention** — e.g. Trash/Spam expiry.
- **Subaddressing** — `user+tag@domain` delivers to `user@domain` with tag available for filtering.

## Webmail client

Built into the DeployWerk SPA; authenticates with **DeployWerk session**; talks to Stalwart via **JMAP** proxied or session-bridged by the API (no separate mail password for browser use).

### Capabilities (phased)

- **Core:** inbox, threads or flat list, read, compose, reply, forward, folders, basic search, attachments.
- **Rich content:** sandboxed HTML, remote images blocked by default, plain-text fallback.
- **Organisation:** labels, rules (mapped to server-side Sieve in later phases), snooze, undo-send window, keyboard shortcuts and mobile gestures.
- **Contacts:** per-team store, import/export vCard, optional CardDAV (later phase).
- **Calendar:** CalDAV, ICS from invites, inline views (later phase).

Full feature parity with consumer mail is not required for early phases; [Phased delivery](#phased-delivery-release-criteria) defines minimum bars.

## Transactional mail API

Teams’ applications send mail through DeployWerk without external ESP dependency.

### Intended HTTP surface (illustrative)

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/api/v1/teams/{team_id}/mail/send` | Single message |
| POST | `/api/v1/teams/{team_id}/mail/send-batch` | Batch |
| GET | `/api/v1/teams/{team_id}/mail/messages/{message_id}` | Delivery status |
| GET | `/api/v1/teams/{team_id}/mail/stats` | Metrics summary |
| GET/POST/DELETE | `.../mail/suppressions` | Suppression list |
| GET/POST/PUT | `.../mail/templates` | Templates |

Exact paths may adjust to match existing API conventions; semantics remain.

### Send request (conceptual)

- **from**, **reply_to**, **to**, **cc**, **bcc**, **subject**
- **template_id** + **variables** (Handlebars-compatible `{{var}}`) and/or **html** / **text**
- **attachments**, **headers**, **tags**
- **idempotency_key**, **scheduled_at**
- Optional **track_opens** / **track_clicks** (privacy-sensitive; off by default for transactional)

### Templates

- Versioned templates; transactional vs marketing categories may have different compliance defaults.
- Plain-text fallback generation when only HTML is supplied.

### Delivery and policy

- States: queued, sent, delivered, bounced, complained, deferred, failed.
- **Hard bounces** → suppression list; API rejects later sends with clear errors.
- **Soft bounces** — retry with backoff.
- **Webhooks** on status transitions (team-configured).
- **Unsubscribe** for marketing: `List-Unsubscribe` / one-click post where applicable.

### Rate limits and quotas

- Per-team daily/hourly caps (entitlements).
- Smoothed sending to protect reputation.
- Dashboard: usage vs limits.

**Dependency:** Rich real-time analytics ideally feeds a **metrics platform** ([05](05-platform-networking-iac-observability.md)). Phase 1 MAY use **PostgreSQL-backed aggregates** until a time-series stack exists; the spec does not mandate a particular TSDB.

## Spam, reputation, and abuse

### Inbound

- Optional **SpamAssassin** / **ClamAV** sidecars (companion managed applications).
- DNSBL checks, DMARC alignment, optional greylisting, quarantine/Junk, Bayesian feedback (phased).

### Outbound

- PTR and DNSBL self-checks; bounce and complaint rate monitoring; optional **warm-up** schedules for new domains; automatic sending pause on threshold breach (configurable).

## Security and governance

- **SMTP AUTH** only over TLS; submission on587 (STARTTLS) / 465 (implicit TLS) as configured.
- **Inbound SMTP:** STARTTLS policy per deployment; configurable strictness.
- **Webmail:** no separate long-lived mail credential in the browser; JMAP authorized via DeployWerk identity.
- **IMAP for external clients:** application-specific passwords (hashed at rest; shown once on creation); revocable.
- **Encryption at rest** for mailbox storage SHOULD align with team KEK / KMS direction in [04](04-security-rbac-secrets-compliance.md).
- **S/MIME and OpenPGP** — import keys, sign/decrypt where supported (later phase).
- **Attachment handling** — flag risky types; optional quarantine policy.
- **Audit:** domain changes, DKIM rotation, mailbox admin, secret reads for mail, transactional sends — recorded per [04](04-security-rbac-secrets-compliance.md).
- **RBAC:** e.g. team admin for mail settings; member for webmail; API token scope such as `mail:send` for transactional API.

## Phased delivery (release criteria)

Phases are **capability gates**, not fixed calendar commitments.

### Phase 1 — Core mail

- Mail domains: registration, DNS wizard, DKIM generation, validation polling.
- Stalwart **`mail_server`** on team destination; preflight; relay mode for blocked port 25.
- Mailboxes and aliases CRUD; personal and service types.
- Webmail: inbox, thread view, compose, reply, forward, folders, basic search.
- Transactional API: single send, basic status and stats.
- Security baseline: TLS for submission, DKIM signing, SPF/DMARC guidance.

### Phase 2 — Delivery reliability

- Bounce/complaint handling, suppressions, threshold-based pause.
- Warm-up schedules; versioned templates (Handlebars).
- Rules UI → Sieve generation; shared mailboxes and send-as; IMAP app passwords; reputation dashboard.

### Phase 3 — Enterprise and advanced

- Contacts + CardDAV; calendar + CalDAV; antispam/antivirus sidecars; mailing lists with subscribe/unsubscribe; S/MIME and OpenPGP; DMARC aggregate ingestion; full metrics integration; LDAP directory sync; multi-mailbox switching in webmail.
